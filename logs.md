Week 4 (A Week) - Focus: Identification of Classes, Objects, System Diagramming

11-5-2026 I added the player GameObject and wrote the first version of PlayerMovement.cs. I used a Rigidbody2D for physics-based movement rather than directly setting Transform.position each frame, which is the correct approach in Unity because moving the transform causes jittery collision, and learning from last Unity project, this was the best way forward. Input is read from the WASD keys each frame and converted into a movement vector which is then applied as a velocity on the Rigidbody2D. I tested the collision with the walls built last week and it worked correctly! The player stops cleanly at the wall boundaries :>. I also added a basic sprinting mechanic multiplying the velocity by a sprintMultiplier constant when Shift is held. The movement felt good to play around with even at this basic stage.

13-5-2026 Today I got the camera following the player and started working on animations. For the camera I wrote a custom follow script using Vector2.SmoothDamp, which gives a nice smooth chase effect rather than the camera just snapping to the player every frame. I also had to add boundary clamping so the camera doesn't show outside the room which required me figuring out the camera's half-width and half-height in world units (using Camera.orthographicSize and the screen aspect ratio), which was more maths than I expected but I understood it once I read a few guides on it :] I had to relearn the animations from my Y10 Unity task, I had to import the sprite sheet, slice the individual frames in Unity's Sprite Editor, then set up an Animator Controller with Idle and Walk states. It was a little more complicated than last time I used Unity, but it came back to me pretty quickly and I didn't have many issues with this. The transition between states is c by a Speed float parameter which I set each frame based on the player's velocity. By the end of the lesson the player was visually running and the camera was smoothly following, so it was a good session overall!

14-5-2026 I came into the lesson with a bug where the player sprite would snap back to its default facing direction the moment you let go of a movement key, after making some changes at home. The issue was that I was updating the facing direction from the player's raw input, but whenthe input is released it becomes zero and the facing resets. The fix was simple enough and I only needed to update the stored facing direction when the input magnitude was above a small threshold, so that releasing a key doesn't interrupt the facing. Small bug, but I was very happy when it was fixed! I also started the shooting mechanic today. The main concept of it is converting the mouse position from the screen space to the world space using Camera.ScreenToWorldPoint, then taking the normalised (normalising is basically scaling vectors) direction vector from the player to the cursor and spawning a projectile in that direction. I did get a basic version firing by the end of the lesson!! Will continue next time.

15-5-2026 after working a bit more at home on it, the projectile from yesterday had two new problems: it wasn't travelling in the right direction, and it was flying straight through the tilemap walls. The direction issue came from the Z component of the mouse world position not being zeroed out. What I learnt is that in a 2D game the camera sits on a different Z plane to the world, so without zeroing the Z, the direction vector was slightly off in a way that made it look really wrong. I learned a new thing there! The wall collision issue was fixed pretty easily to be honest by adding a LayerMask to the projectile's trigger so it only registers hits on the correct physics layer. Unity's layer system is something I hadn't used properly before so this was good practice. By the end of the lesson shooting felt clean and responsive which was very satisfying!

Week 7 (B Week) - Focus: Programming, Asset Creation/Identification and Journaling

1-6-2026 - This was a big day! I made the big decision to change the project direction from the dungeon crawler / Echo puzzle concept to a horde-survivor roguelike. After thinking about it over my days off I realised that the roguelike format would showcase OOP concepts much more clearly, as wave-based enemies, a gun inheritance hierarchy, an ability system with polymorphism all works well in the OOPy format. I still kept the Echo mechanic but I made plans about moving it to being an unlockable ability rather than the main mechanic. I'm happy with the call, as I think it is more feasable within this time. To get started with this, I started building Enemy.cs as an abstract class with the shared fields (currentHealth, maxHealth, rb, spriteRenderer, player reference) and shared methods (TakeDamage(), MoveTowardPlayer(), Die()). I then created BasicEnemy as the first subclass which just inherits everything without changing anything. By the end, a BasicEnemy was chasing the player around the room and dying to bullets! Very happy >:D


3-6-2026 - Another (semi?)productive double today. I built the weapon system. Gun.cs is abstract and contains all the firing logic like cooldown tracking, magazine management, reloading and recoil. CursorGun is the main subclass and only adds GetAimDirection(), which is an abstract method in the parent, showing how a subclass can add on behaviour without changing the shared mechanics. I also created GunProfile as a ScriptableObject so I can define weapons as data assets rather than hardcoded values which means creating a new weapon is as simple as just making a new asset in the editor, which is much more scalable IMO. I also started the weapon draft manager on the side. I had some bugs but that's something to fix next lesson.


5-6-2026 - I fixed the bugs from last lesson at home. The main one was that the shotgun was only shooting one projectile when in the code it stated it was shooting multiple :/ After some investigation 🧐🔎 I found that in ApplyProfile(), the base stats are read from the profile first, then all stored upgrades are re-applied on top. The problem was that pelletCountBonus and spreadAngleDelta were not being reset before the profile was applied, so they were doubling up with each call, meaning a second star upgrade was adding pellets on top of already-boosted values. The fix was resetting them both to zero at the start of ApplyProfile() before reading from the profile. A good bug to crack 😀

Also ran into a separate issue after getting the ammo UI (UIAmmo.cs) compiling. It threw three errors straight away!! Two were CS0103: The name 'Mouse' / 'Keyboard' does not exist, this is because I forgot to add using UnityEngine.InputSystem; at the top of the file, which was very stupid. The third was a warning about FindFirstObjectByType being deprecated, I searched it up and fixed it by swapping it to FindAnyObjectByType which is the current recommended Unity call. Small errors but good lessons learnt about keeping track of which packages your code uses. By the end, the ammo counter was showing correctly in the HUD and flashing red on empty!

Bonus problem for all you fans: I noticed the gun sprite wasn't tracking the cursor at all and it was always pointing the same direction regardless of where I aimed. Found that in AimAtCursor(), I was calculating the correct angle but never actually applying it to aimPivot.localRotation, I'd stored it in a variable and forgotten to use it 🥲. Fixed by getting the pivotAngle from aimRotation and assigning it to the pivot's localRotation each frame. The gun now properly tracks the cursor! This one caused me lots of headaches.

Week 8 (A Week) - Focus: Programming, Journaling and Testing and Evaluating

8-6-2026 - I built the progression system today! I started it at home really but just kept at it today. PlayerProgression.cs tracks the current exp, current level and the threshold for the next level-up, which I made scale exponentially so that early levels feel fast but later ones require more grinding. It fires a OnLevelUp C# event that other systems listen out for. ExpOrb.cs is the collectible and it sits until the player enters its attract radius, then ZOOMS toward the player each frame using a speed multiplier, and calls AddExp() on collection. LevelUpDisplay responds to the OnLevelUp event by pausing time and showing the upgrade selection screen. 

Also a remaining gun flip issue from last week. When the cursor moved to the left side of the screen, the gun sprite was flipping using flipY correctly but the pivot position wasn't moving, so the gun looked like it was held behind the player. The fix was to mirror the local X position when facing left, using the movement direction to determine which way the player is facing. I also connected this to PlayerAnimationDriver.cs so the player sprite flips to match, by exposing a SetAimFlip(bool flipped) method on the driver that the gun calls every frame from AimAtCursor(). This required some back-and-forth because aiming directly up or down was making the player sprite flicker rapidly between left and right. I resolved this by only flipping when the cursor has a meaningful horizontal offset from the player by adding a deadzone to the direction.x comparison so directly vertical aiming doesn't trigger either side :}
10-6-2026 Today i set up the ability framework and the wave spawner. Its the same interface but completely different behaviours which is polymorphism working as polymorphism does. HordeSpawner manages wave timing and picks enemy types from a pool, with the health and speed scaled by the player's current level. It's working well, but something feels off about the level scaling. I need to figure that out, probably not even next time.
